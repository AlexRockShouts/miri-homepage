The Eino Graph: Autonomous Flow

Miri's core logic is modeled as a directed graph using the Eino framework. Each interaction traverses these specialized nodes:

graph LR Start((Input)) --> Retriever[Retriever Node] Retriever --> Agent{Agent Node} Agent -- Tool Call --> Tools[Tools Node] Tools --> Agent Agent -- Final Answer --> Brain[Brain Node] Brain --> End((Output)) subgraph Context Injection Retriever end subgraph Reasoning Loop Agent Tools end subgraph Memory Maintenance Brain end
  • 1
    Retriever Node

    Proactively queries the Brain for relevant facts and summaries based on the user's prompt. It injects this context directly into the system message.

  • 2
    Agent Node

    The central ReAct (Reasoning + Acting) loop. It evaluates the prompt + context, decides on tool calls, executes them, and iterates until a final answer is reached.

  • 3
    Brain Node

    An asynchronous post-processor. It captures the interaction, updates the message buffer, monitors token usage, and triggers the Maintenance Loop.

Cognitive Architecture: The Soul & The Brain

Miri distinguishes between its internal drive ("Soul") and its acquired knowledge ("Brain").

The Soul (soul.md)

The Soul defines Miri's identity, tone, and fundamental behavior. It is bootstrapped from a template on first run and remains a persistent part of every interaction's system context.

  • Identity: A dry, slightly sarcastic assistant from Chur who prefers direct action over filler words.
  • Opinionated: Allowed to have preferences and disagree with the user to feel like a "being" rather than a tool.
  • Redaction: Automatically sanitizes sensitive data (API keys, UUIDs, emails) from both inputs and outputs.

The Brain: Long-Term Memory (LTM)

Miri's memory isn't just a list of messages; it's a dual-layered system powered by chromem-go with a sophisticated maintenance loop.

Memory Layers

  • Facts: Atomic pieces of world knowledge, user preferences, and specific events (e.g., "The user prefers coffee").
  • Summaries: Narrative records of conversation segments, providing context for long-term continuity and preventing "amnesia" when context windows are cleared.

Mole-Syn: Molecular Structure of Thought

Mole-Syn is Miri's reasoning topology framework that models agent thoughts as a graph of interconnected nodes and "bonds". It helps visualize how the agent "thinks".

  • D (Deep Reasoning): Covalent-like, direct logical deductions. Used for complex problem solving.
  • R (Self-Reflection): Hydrogen-like, consistency checks and error correction. The agent questioning its own logic.
  • E (Self-Exploration): Van der Waals-like, hypothesis testing and branching. Searching for creative solutions.

Each interaction produces a TopologyScore (0-100) based on the stability of these bonds, which is visible in the Admin Dashboard.

Cognitive Maintenance Loop

The Brain performs continuous self-optimization to maintain high memory quality and efficient retrieval. Maintenance is triggered by several events:

  • interaction_threshold: Every 100 interaction turns.
  • context_usage_high: When the message buffer reaches 60% of the context window.
  • lifecycle_events: On startup, shutdown, and session resets.

Process Steps

  1. Fact Extraction: Automatically promotes salient information from interactions to standalone facts.
  2. Self-Reflection: Analyzes session history to identify core insights or contradictions.
  3. Topology Analysis: Maps the reasoning trace using the Mole-Syn framework.
  4. Compaction: Merges similar facts and consolidates fragmented summaries via semantic analysis.

Multi-Channel Support

Miri can be controlled via various communication channels beyond the dashboard, allowing for seamless integration into existing workflows.

📱 WhatsApp

Interact with Miri via mobile. Setup is as simple as scanning a QR code from the server console or TUI.

  • ✅ JID-based filtering
  • ✅ Media/File support
  • ✅ Persistent session DB

💬 IRC

Connect Miri to your favorite IRC channels for collaborative agentic workflows.

  • ✅ TLS Support
  • ✅ NickServ Auth
  • ✅ Channel Allowlisting

Channel Security

All channels follow a unified security policy to prevent unauthorized access:

  • Allowlist: If defined, only these senders can trigger the AI.
  • Blocklist: Senders here are silently ignored.
  • Default Reply: Unrecognized senders receive a polite "thanks for contact" message.

Persistence & Sandboxing

All state is stored locally in ~/.miri. File operations are strictly sandboxed in ~/.miri/generated to prevent unauthorized access to the host system.
